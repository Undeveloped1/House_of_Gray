# Cursor Methodology Report Pattern

When Paul and Joe develop a design methodology during a session that Cursor needs to follow, write a self-contained methodology report for Cursor's repo.

## When to write

- A new naming convention is established
- A card design pipeline step is refined
- A quality gate or review filter is created
- A "how to think about X" methodology emerges from discussion

## Report structure

1. **Pipeline order** — where this step fits in the full design pipeline
2. **The method itself** — step-by-step, with examples
3. **Categories/rules** — what patterns to use and when
4. **Examples** — real cards from the canonical roster with art direction
5. **What NOT to do** — anti-patterns with explanations
6. **Reference roster** — pointer to the canonical roster file

## What NOT to include

- Session narrative ("Joe said," "we discussed")
- Design philosophy that isn't actionable for Cursor
- Open questions or deferred items
- Anything Cursor can't act on directly

## Target audience

Cursor agent reading CARD_CRAFTING_METHODOLOGY.md. Language should be instructional, not conversational. Every section should answer "how do I apply this when I'm drafting a card?"

## File placement

Write to `design/Cursor_<Topic>_Report_YYYY-MM-DD.md` in the vault. Cursor pulls from vault or receives the report via bridge.
