# Git Recovery Pattern — Stubbed Source Files

When a workspace source file has been reduced to a stub (e.g., "MERGED — DO NOT EDIT THIS FILE"), recover the original body from git history.

## Steps

1. Find commits that touched the file:
   ```
   git log --oneline -5 -- "docs/Paul/workspace/<filename>"
   ```

2. Pick a commit before the file was stubbed. The stub commit is usually the most recent — go one commit back.

3. Show the original content:
   ```
   git show <commit>:"docs/Paul/workspace/<filename>"
   ```

4. If the file is large, pipe to head/tail to read sections:
   ```
   git show <commit>:"<path>" | head -80
   git show <commit>:"<path>" | tail -n +80
   ```

## Example — Spells v6 Draft

The workspace file `Bruiser_Spells_v6_Draft_2026-05-30_Paul.md` was stubbed to a 15-line redirect. The original body (200+ lines with spell concepts, moments, verdicts, and design methodology) was recovered from commit `982df70`:

```
git show 982df70:"docs/Paul/workspace/Bruiser_Spells_v6_Draft_2026-05-30_Paul.md" | head -80
```

The recovered body contained:
- 3 LOCKED spells with creative moments
- 7 PROPOSED spells with scene prose
- 3 NEEDS REWORK concepts
- 8-step spell design methodology
- Full verdicts with design reasoning

The names had already been banked in Card_Bank.md during DCW Batch 0, but the **spell moments** (creative scene prose) were never captured. This gap was only discoverable by recovering the original body — the stub contained no hint of what was lost.

## When to use

Any time a source file says "MERGED — DO NOT EDIT," "SUPERSEDED," or is suspiciously short (10-20 lines) while the DCW queue or recovery plan says it was a substantial working document. The stub preserves the pointer; git preserves the body.
